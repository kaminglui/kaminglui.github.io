In plain English — what is reinforcement learning?

Imagine training a puppy. You don't write instructions — you let the puppy try things, and you hand out treats when it does something right. Over time, the puppy figures out what behaviour earns treats and does more of it. That is reinforcement learning: an agent (the puppy) takes actions in an environment (your living room), gets rewards (treats), and updates its policy (its habits) to earn more treats over time.

The bandit below is the puppy version of this — there's no living room, no states, just a row of buttons. Press one, get a treat or not, and remember how often each button pays. The question is: how do you tell the good buttons from the unlucky ones without wasting too many tries?

Words that show up a lot:

  • Agent — the learner that makes decisions (the puppy, the bandit player).
  • Environment — the world the agent lives in; it responds with rewards and next states.
  • Action — a choice the agent makes (pull arm 3, move left, accelerate).
  • Reward — a number the environment hands back after an action. Positive = good, negative = bad.
  • Policy (π) — the agent's strategy: a rule mapping situations to actions.
  • Return — total reward the agent collects, usually future rewards discounted by γ.
  • Exploration vs exploitation — try new things to learn (explore), or cash in on what you already know (exploit). The core RL dilemma.

RL theory tour

1 · Bandits: RL without state

Kid version. A bandit is a row of gumball machines. You don't know which one has the most candy, so you try each a few times, then mostly use the best one — but sometimes you still try the others in case you were wrong.

A bandit has k arms. Pulling arm a returns a random reward drawn from an unknown distribution with mean μa and (optionally) std σa. The agent keeps a running estimate Q(a) of μa and a pull count N(a), updated incrementally:

  • Q(a) — running mean-reward estimate for arm a (starts at 0).
  • rt — the reward just observed after pulling a.
  • N(a) — number of times arm a has been pulled.

Shape watch. is the update you'll see again in TD, SARSA, Q-learning — the step size α and the target change, but the skeleton "nudge the estimate toward a target" stays the same.

Three rules for picking the next arm:

ε-greedy

  • ε ∈ [0, 1] — the probability of exploring (picking a random arm). Complement 1 − ε is the probability of exploiting (picking the current best-estimated arm).
  • ε = 0 is pure exploitation — never tries a new arm, will freeze on whatever looked best after the first pulls. ε = 1 is pure exploration — ignores everything learned. 0.05–0.1 is typical; "decay ε over time" is common.
  • Simple, works, but wastes ε on known-bad arms forever.

UCB1 (optimism)

  • c — how much to trust the uncertainty bonus (≈ 1–2).
  • Bonus is large when N(a) is small → uncertain arms get tried.
  • Regret is O(log t); deterministic once seeded.

Thompson sampling

  • Draw one sample from each arm's posterior over μ, pick the highest.
  • Explores more when posteriors are wide, less when tight.
  • Bayesian, strong empirically, naturally randomised.

Demo — Bayesian coin: prior · likelihood · posterior

Kid version. You have a coin; you're not sure it's fair. You start with a guess of how biased it is (that's the prior). You flip it and watch how heads/tails come out (likelihood — how probable the data is for each possible bias). You update your guess to account for what you saw (posterior). Repeat. The more flips, the sharper your belief.

Bayes' rule — the update machinery behind Thompson sampling:

  • θ — unknown parameter (here, the coin's head-probability p ∈ [0, 1]).
  • 𝒟 — observed data (here, the sequence of h heads and t tails).
  • Prior p(θ) — belief before seeing data. Is a probability distribution (integrates to 1).
  • Likelihood p(𝒟|θ) — how probable the data is, viewed as a function of θ. Is not a probability distribution over θ — doesn't integrate to 1; in fact, it has no units of "density over θ" at all.
  • Evidence p(𝒟) — marginal probability of the data; just the constant that renormalises the posterior.

Why bandits matter for full RL: every RL algorithm still has to solve which action next?. Bandit ideas (ε-greedy, softmax, optimism, posterior sampling) get reused inside Q-learning, SARSA, actor-critic to pick actions at each state.

Is the bandit the "root" of RL? Partially. It's the simplest setting that isolates the exploration/exploitation trade-off. But full RL adds state and long-horizon credit assignment — problems bandits ignore. So: the root of the exploration half, not the whole tree.

2 · MDP: adding state and time

Kid version. A bandit is one room with buttons. An MDP is a whole house with doors — the button you press decides which room you end up in next, and different rooms have different buttons. Now you have to plan: "should I grab the candy in this room, or walk to the room with even better candy?"

A Markov Decision Process is a 5-tuple:

  • 𝒮 — set of states the environment can be in (e.g. which room).
  • 𝒜 — set of actions the agent can take (e.g. left / stay / right).
  • P(s'|s, a) — transition probability into s' after doing a in s. Markov property: this depends only on the current (s, a), not on history.
  • R(s, a) — expected immediate reward for doing a in s.
  • γ ∈ [0, 1] — discount. γ ≈ 0 is myopic ("I want candy now"), γ → 1 is patient ("more candy later is fine").

The agent's goal: maximise the expected return — the discounted sum of future rewards:

A policy is the agent's strategy: a probability of picking action a in state s.

3 · Value functions: how good is this?

Kid version. Imagine grading every room in the house ("this room is worth 7 candies over the rest of the afternoon"). The room's grade is its state-value V(s). Grading each action from a room ("opening this door is worth 9") is its action-value Q(s, a).

Three value functions. All three are expectations of the future return Gt under policy π:

  • Vπ(s) — "starting from s and following π, how much total reward do I expect?"
  • Qπ(s, a) — "starting from s, taking a first, then following π."
  • Aπ(s, a) — the advantage: how much better than average is action a? Baseline-subtracted, lower-variance than Q.

4 · Dynamic programming: when you know the model

Kid version. If someone hands you the rulebook of the game (every door and every reward written down), you can sit at a desk and just solve it. No playing required. That's dynamic programming.

When P and R are known, two classical algorithms solve for V* and an optimal π. Both are fixed-point iterations on the Bellman operator; they're duals of each other.

  • Rπ — vector of expected immediate rewards under π: Rπ(s) = Σa π(a|s) R(s, a).
  • Pπ — transition matrix under π: Pπ(s, s') = Σa π(a|s) P(s'|s, a).
  • Δmax — worst-case change across states in one sweep; the contraction metric.
Policy iteration Value iteration
Policy evaluation. Fix π, iterate until V stabilises. One-step sweep.
Policy improvement. . Repeat eval → improve until π stops changing. Repeat sweep until is small. Extract π as the greedy policy at the end.
Fewer outer iterations, each one evaluates a whole policy. More outer iterations, each is cheap — fuses evaluation + improvement.

Both converge to V* because the Bellman operator is a γ-contraction — each sweep shrinks the worst-case error by factor γ, so convergence is geometric: after n sweeps the residual is at most γn Rmax / (1 − γ). The 1D-chain demo below plots this contraction live.

Both assume you can read P and R — which is almost never true in practice. The rest of the tour (§5–§7) is how to learn without the rulebook.

5 · Model-free: Monte Carlo vs temporal difference

Kid version. You don't have the rulebook, so you play the game a lot and learn from what actually happens. Monte Carlo: play a whole game, write down the final score, update. TD: after every move, look at the next square and update your guess a tiny bit — don't wait for the game to end.

Two ways to learn Vπ from samples of experience (st, at, rt+1, st+1) — no P and R needed:

Monte Carlo (MC)

  • Gt — the actual return from the completed episode.
  • Unbiased, high variance; needs the episode to terminate.

TD(0)

  • Bootstrap: replace Gt with rt+1 + γ V(st+1).
  • Biased early, low variance, works online.

n-step / TD(λ)

  • Interpolates MC (n = ∞) and TD(0) (n = 1).
  • TD(λ) averages every n with weight (1−λ)λn-1.

The TD error — the core unit of bootstrap learning:

  • δt — "surprise": how wrong my prediction was once I saw the next step.
  • α — step size (learning rate) ∈ (0, 1].
  • γ — same discount as before.

6 · On-policy vs off-policy, and importance sampling

Kid version. On-policy: learn from your own games, the ones you played with your current strategy. Off-policy: learn from someone else's games (your old self, a replay of a friend, a random walker). Watching others is cheaper — but you have to re-weight what you see, because their choices weren't your choices.

SARSA and Q-learning are the textbook pair — the only difference is the target:

  • at+1 (SARSA) — the action the current policy actually took next. Learns Q for whatever behaviour (ε-greedy, noisy) is being run.
  • maxa' (Q-learning) — the value of a different, greedy policy. Learns Q* regardless of what action was taken.
On-policy (SARSA)Off-policy (Q-learning)
Data source Own trajectories from π Any behaviour policy μ — replay buffers, demonstrations
Trade-off Safer during exploration (learns value of ε-greedy itself) Learns the optimal policy directly; can reuse old data

When the data came from a different policy μ than the one we're evaluating π, rewards are weighted incorrectly. Importance sampling corrects this by reweighting each step by ρ:

  • π(a|s) — target policy (the one we want to evaluate).
  • μ(a|s) — behaviour policy (the one that actually generated the data).
  • ρ(τ) — product of per-step ratios over the trajectory τ.

Unbiased but high-variance (the product over a long trajectory can explode). Weighted importance sampling (normalise by Σ ρ) is biased but much lower variance — practical off-policy methods lean on it. Q-learning sidesteps IS entirely by using a max that doesn't depend on which action was actually taken.

7 · Policy gradients and actor-critic

Kid version. Instead of grading every room, just tweak your habits directly. Did that habit lead to good outcomes? Do more of it. Bad? Do less. Actor-critic combines both: the actor is the habit; the critic is the grader that tells the actor whether it did better or worse than expected.

Value-based methods (Q-learning, DQN) learn Q, then act greedily. Policy-based methods skip Q and directly parameterise the policy . The policy gradient theorem:

  • θ — parameters of the policy (actor) network πθ.
  • φ — parameters of the critic network Vφ or Qφ (actor-critic only).
  • J(θ) — expected return under πθ (what we want to maximise).
  • ∇θ log πθ(a|s) — "score function": which direction to nudge θ to make action a more likely at state s.
  • Ψt — a signal saying "was this good?". Lower variance = better choice.

Gradient-ascent step (drop this into SGD/Adam):

REINFORCE

  • Pure Monte Carlo policy gradient.
  • Unbiased, very high variance.

Actor-Critic (A2C)

  • Critic learns Vφ; actor ascends along δt.
  • Lower variance, small bias, online.

PPO / TRPO

  • Actor-critic with a clipped ratio / trust region.
  • Keeps policy updates small and stable.
  • GAE(λ) — Generalised Advantage Estimation: exponentially-weighted average of n-step TD advantages Σ (γλ)k δt+k. λ=0 is TD(0); λ=1 is Monte Carlo.

Deep off-policy actor-critic cousins — DDPG, TD3, SAC — use the same skeleton with different tricks for stability and exploration.

8 · Monte Carlo Tree Search — planning with a model

Kid version. Imagine a chess engine. It doesn't know the optimal move, but it can imagine playing out a game — "if I move here, and then this, and then that…" — and track which branches tend to end well. After doing that thousands of times it has a little tree in its head, with the best-looking branch explored most. Monte Carlo Tree Search (MCTS) is that idea, made rigorous. Every branch carries two counters: how many times it's been tried (N), and the average reward when it has (Q). A tiny exploration bonus nudges the engine to try underexplored branches occasionally so it doesn't lock onto an early lucky streak.

Everything upstream of this section — DP, TD, policy gradients — learns from experience without planning ahead. MCTS is the other axis: it's a planning algorithm that sits on top of a simulator. You don't need the full transition table that value iteration needs (§4); you only need something that, given (s, a), produces (s', r). MCTS grows a search tree rooted at the current state and uses it to pick the next action.

The four-stage loop

  • Select. Start at the root. Recursively pick the child with the highest UCT score (see below) until you reach a node that isn't fully expanded.
  • Expand. Add one of its unvisited children to the tree.
  • Rollout (a.k.a. simulation). From that new node, play a default policy — often uniform-random — until the game ends (or a fixed depth). Call the observed return G.
  • Backup. Walk back up the tree you descended; at each node increment N by 1 and add G to W. The per-node estimate Q = W / N is the running mean of returns through that node.

UCT — the selection rule

"Highest UCT score" means Upper Confidence bound for Trees (Kocsis & Szepesvári, 2006):

Same flavour as the UCB1 bandit formula from §1, with two differences: Q is the value through that child (mean of backed-up returns), not an instant-reward estimate; and it's applied recursively at every level of the tree. An unvisited action has N(s, a) = 0 so UCT = ∞ — every action gets tried at least once before UCT starts discriminating. c trades exploration against exploitation; c = √2 ≈ 1.41 is common for rewards in [0, 1].

AlphaZero — MCTS + a neural network

Classical MCTS has one annoying property: the rollout phase is noisy. Random play rarely captures strong lines in large games (chess, Go), so you need millions of simulations to get a usable estimate at a leaf. AlphaZero swaps the rollout for a value-network estimate and biases the select phase with a policy prior . The UCT variant it uses is PUCT:

Training loop: self-play with MCTS-guided moves, record visit-count distributions at each move, train the policy network to match those distributions and the value network to predict the game outcome. Same four-stage loop; smarter signals at select and rollout; no hand-crafted evaluation functions. This is the backbone behind AlphaGo Zero, AlphaZero, MuZero, and the LLM-guided theorem provers and code-search systems that came after.

Where MCTS sits in the RL landscape: DP (§4) is planning when you have the table. MCTS is planning when you only have a simulator. Model-free methods (§5–7) are learning when you don't even have that, just experience. Modern systems routinely combine all three — learn a model from experience, plan with MCTS in the model, distil the plan into a policy network. The "shape" is the same update-toward-a-target loop covered next (§9).
